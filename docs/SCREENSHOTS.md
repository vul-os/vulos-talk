# Screenshots

The images in [`docs/screenshots/`](screenshots/) are generated by a Playwright
script against a **fully seeded demo backend** — no real accounts, no cloud, and
no manual setup.

## Regenerate

```sh
npm install
npx playwright install chromium   # one-time: download the headless browser
npm run screenshots
```

`npm run screenshots` (scripts/screenshots.mjs):

1. Builds the frontend (`dist/`) if missing and the Go binary.
2. Starts the binary on port `8083` with **auth enabled** and a throwaway data
   dir (`/tmp/vulos-talk-demo-data`), using a fixed demo JWT secret.
3. Seeds Spaces channels, a threaded multi-author conversation, reactions, a
   pinned message, a DM, and scheduled huddles via the REST API
   (scripts/seed-demo.mjs).
4. Injects a demo `session` cookie and captures the screens at 1440×900 (dark).
5. Stops the server and writes a manifest to `docs/screenshots/README.md`.

### Against a live instance

```sh
BASE_URL=https://talk.example.com npm run screenshots
```

When `BASE_URL` is set the script captures the live instance instead of starting
its own server (it still injects the demo session cookie, so point it at a
non-production instance).

## Why auth-enabled seeding?

The backend derives a message's author from the verified JWT **subject**, never
from a client header (a deliberate anti-forgery measure). To produce a realistic
**multi-author** conversation, the seeder runs the backend with auth enabled and
mints a short-lived HS256 token per demo author (`scripts/seed-demo.mjs`,
`mintToken`), sending each message with that author's bearer token. This is the
only honest way to attribute seeded messages to different people.

## Captured screens

| File | Surface |
|------|---------|
| `hero.png` | Spaces — `#general` with the seeded team conversation (reactions, pin, member roster) |
| `thread.png` | `#engineering` with the threaded reply side-panel open |
| `dm.png` | A direct message between two teammates |
| `search.png` | In-channel search with live, highlighted results |
| `slash-command.png` | Slash-command autocomplete in the composer |
| `quick-switcher.png` | The command-palette quick switcher (⌘K) with match highlighting |
| `bots.png` | Apps & Bots — create and manage bot apps + tokens |
| `mobile.png` | Single-column mobile layout with the bottom nav (390×844) |
| `huddles.png` | The huddles dashboard — scheduled rooms with join links |

## Limitations

Live **huddle video** (WebRTC media) and real-time **websocket presence** cannot
be reproduced headlessly, so `huddles.png` shows the scheduling dashboard rather
than an in-call view, and presence reflects only the demo screenshot session.
Everything else is real, seeded data served by the Go binary.
